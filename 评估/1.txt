## 1. 现在的结论：Dense 主力，Hybrid 在调到合适配置后“至少不拉跨”

### 第二组（池子=10）的情况是关键

全局（top_k=8）：

- **BM25**：Hit 66.7%，MRR 0.373
- **Dense**：Hit 94.4%，MRR 0.673
- **Hybrid**：Hit 94.4%，MRR 0.540

按类型拆：

- **exact_keyword**（5个）
    
    - Dense: 100% / 0.900
    - BM25: 100% / 0.690
    - Hybrid: 100% / 0.750
    
    → 三者都稳，Dense 最好，Hybrid 也不差。
    
- **chinese_semantic**（5个）
    
    - Dense: 80% / 0.467
    - BM25: 20% / 0.033
    - Hybrid: 80% / 0.197
    
    → Dense > Hybrid > BM25。  
    Hybrid Hit 跟 Dense 一样，但 Rank 明显没 Dense 好（MRR 低），说明排序上还是 Dense 稳。
    
- **english_semantic**（3个）
    
    - Dense: 100% / 0.733
    - BM25: 66.7% / 0.381
    - Hybrid: 100% / 0.456
    
    → 同样是 Dense 最好，Hybrid 命中率跟上，但 MRR 稍差。
    
- **mixed**（5个）
    
    - Dense: 100% / 0.617
    - BM25: 80% / 0.392
    - Hybrid: 100% / 0.740
    
    → 这里 Hybrid 排名反而略优（MRR 最高），说明在这部分 query 上 BM25 给了有用的信号。
    

### 第一组（池子=20）可以看成对照

全局：Hybrid 只有 72% 命中，比 Dense 差一截。  
说明 pool_size 放太大，在你这个小语料上，BM25 噪声更容易挤掉 Dense 的好结果。

所以你“总算知道了”：

- pool_size 太小：Hybrid 几乎变成 Dense-only；
- pool_size 太大：BM25 噪声太多，拉垮 Dense；
- 在你目前的语料 & query 分布下，**pool_size≈10** 是 Hybrid 比较平衡的点。

## 2. 这对你“怎么选引擎 / 怎么讲项目”意味着什么？

### 结论 1：Dense-only 可以放心当主力引擎

现在你有真实数据支撑：

- Dense 在所有 query 类型上，Hit@K 都最高或并列最高；
- MRR 也始终领先 BM25/Hybrid，说明它不仅命中，而且排得更靠前。

你可以在项目里明确说：

> “在我这个 DevAgent 项目里，经过检索评估，Dense-only 是最稳的主力检索方式，我们把它作为默认引擎。”

### 结论 2：BM25/Hybrid 的作用是“增强、兜底”，不是取代 Dense

- BM25：
    
    - 在 exact_keyword / mixed 上有明显价值（特别是带 `.py`、函数名的 query）。
    - 在中文语义上几乎废，英文语义上一般。
- Hybrid（pool_size=10、RRF + 权重适当）：
    
    - 全局 Hit 跟 Dense 持平（94.4%）；
    - 在混合类 query 上 MRR 甚至略好；
    - 在中文/英文语义类 query 上，命中率跟 Dense 一样，排序略差。

这就给你一个很自然、也很合理的设计：

> - 默认用 Dense-only；
> - 对“包含 `.py` / `def xxx` / 明显符号”的 query，尝试用 Hybrid 或 BM25 做辅助；
> - 或者只在某些工具（比如 AST 函数抽取失败时）用 Hybrid/BM25 做兜底。

你不需要把 Hybrid 硬塞成“全局默认策略”，因为 Dense 本身已经非常顶了。
